## Pre-registration (verbatim, from an arm's `config.json` `description`)

> PHASE 21 SCENARIO exploration_recovery (package A; Tier-4 acceptance 3: exploration recovers neglected interests in at least one designed scenario). This committed config IS the CONTROL arm (exploit-heavy world, exploration.enabled=true but epsilon=0 throughout -> a permanent filter bubble); scripts/run_phase21_a.sh patches it into the two recovery arms. All three arms share seed 42 and the identical generated world; exploration.enabled=true in EVERY arm so the ExplorationCandidateSource is registered and its per-slot bernoulli draws are consumed identically (stream-safe, P21-CONTRACTS section 1). HYPOTHESIS: in a neglected-interest world produced by an exploit-heavy policy, switching exploration on partway through the run (exploration.enable_at_day=4) recovers neglected-CHANNEL exposure (tail/small creators and off-preference content) relative to a never-exploring control on the same world, and MORE exploration recovers more (dose-response epsilon 0.10 >= 0.05). MECHANISM: all three arms use a similarity-dominant exploit ranking (similarity_weight 0.95, every other novelty/diversity signal + repetition penalty zeroed) PLUS exploration_weight=1.0 in this shared base -- exploration_weight makes exploration-SOURCED candidates visible in the ranked feed WHEN they exist, instead of being buried by the similarity term; it is INERT in the epsilon=0 control and in every arm's pre-gate window (no exploration candidates are produced there), so the control stays a pure exploit bubble and days 0-3 are byte-identical across all arms. (A reduced-scale pilot -- the committed tests/property/exploration_recovery_statistical_test.cpp -- established that with exploration_weight=0 the fired exploration content never reaches impressions, so exploration_weight>0 is required for the intervention to do anything; the same pilot also found that mean_preference_entropy does NOT fall under the exploit bubble here -- the Phase-20 saturation/aversion dynamics make hidden preferences WANDER, so entropy can drift UP in every arm and is reported but is NOT the primary recovery signal.) Before the gate day the EFFECTIVE epsilon is 0 in every arm (the gate forces bernoulli(0), consuming the same uniform01 as bernoulli(epsilon)); at enable_at_day=4 the recover arms fire exploration (underexposed / random-fresh / uncertain-topic modes) so neglected tail creators + off-preference content reach users. EXPECTED SIGNATURE: (validity) each recover arm's day 0-3 ecosystem_metrics.csv + longterm rows EQUAL control's exactly (strong stream-alignment check); (recovery) after day 4, tail_creator_share of the recover arms rises ABOVE control and creator_hhi falls BELOW control (exploration de-concentrates impressions toward neglected creators), the served archetype mix broadens away from the single dominant archetype, and recover10 >= recover05 (dose-response). mean_preference_entropy / mean_preference_shift are reported for completeness but with the wander caveat above. VERDICT CRITERIA: recovery CONFIRMED if the days 0-3 validity match holds AND post-gate tail_creator_share(recover05) exceeds control by a clear margin AND creator_hhi(recover05) falls below control AND recover10 is at least as strong. Report 'effect not observed' honestly if tail/hhi do not move. ARMS: control = THIS config (algorithm hnsw_ranker_diversity COMPLETE mode; exploration.enabled=true, epsilon=0.0, enable_at_day=-1; exploration_weight=1.0 inert here; exploit ranking below). recover05 = epsilon=0.05, enable_at_day=4. recover10 = epsilon=0.10, enable_at_day=4. Shared exploit ranking = {similarity_weight:0.95, quality_weight:0.0, freshness_weight:0.0, popularity_weight:0.0, trending_weight:0.0, creator_affinity_weight:0.0, exploration_weight:1.0, repetition_penalty:0.0} -- same similarity-dominant exploit weights as the filter_bubble bubble arm (which sets exploration_weight 0, the only ranking difference; both are inert-exploration in their epsilon=0 states), so this scenario studies recovery FROM that bubble. Pre-registered before any run per P21-CONTRACTS section 4.

## Notes

- **No aggregate score** (D22 / V2 TDD §6): every group above (engagement, hidden welfare, session health, recommendation quality, long-term, ecosystem) is reported SEPARATELY; this script never combines them into one number, and neither do plot_results.py's Phase 21 additions.

- **Frozen-schema reads only** (contracts §6): every key this script reads is quoted verbatim from contracts §2 (ecosystem) / §3 (longterm_metrics.csv's trailing column) / §5 (long_term, inherited from Phase 20) / the results_writer.cpp-verified V2 §6 group keys -- no candidate-path guessing. A block genuinely absent for an arm (e.g. a control arm run with a different gate set) renders as a clearly labeled n/a cell, never a crash or a silently-guessed alternate key.

- **Per-day mini-table method**: first/middle/final SIMULATED DAY over the UNION of days present in `ecosystem_metrics.csv` and `longterm_metrics.csv` (the two files are gated independently -- contracts §1 requires the event scheduler for the former, §5's P20 gates for the latter -- so an arm with only one of them still gets a partial row).

- **CSV/MD content split**: `<scenario>-comparison.csv` carries only the flat whole-run arms-x-groups table (one row per arm); the per-day mini-table, the verbatim pre-registration quote, and the verdict are `.md`-only (see this script's module docstring).

- **Concurrency-contention caveat**: if arms were run CONCURRENTLY (contracts §4 caps this at 2 simulate processes per package), wall-clock/timing numbers carry cache and memory-bandwidth contention; every other number in this report is deterministic (rng/clock-free, D8/D9) and unaffected by run mode.

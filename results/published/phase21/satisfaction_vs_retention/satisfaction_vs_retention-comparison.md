## Pre-registration (verbatim, from an arm's `config.json` `description`)

> PHASE 21 SCENARIO — SATISFACTION-VS-RETENTION FRONTIER (Package B; V2 TDD s4.18 / s10 item 7; contracts s4). Base: configs/realism-medium-retention.json + evaluation.ecosystem_metrics=true, seed 42, algorithm hnsw_ranker, event mode, DEFAULT archetype catalog. FIVE arms on ONE identical world; only ranking weights differ.
>
> HYPOTHESIS: a retention/welfare FRONTIER exists — sweeping ranking weight from the engagement preset (l=0) to the satisfaction-proxy preset (l=1) trades engagement-side outcomes (retention_7d, sessions/user/day) against welfare-side outcomes (mean hidden satisfaction, session utility U_s). The engagement end maximizes retention/time-on-app; the proxy end maximizes hidden welfare; comfort-content dynamics surface in archetype shares (arch_comfort).
>
> MECHANISM / BLEND RULE: ranking weights are linearly interpolated per-key over the UNION of the engagement and proxy patch keys (17 keys) at l in {0, 0.25, 0.5, 0.75, 1}. For each union key, value(l) = (1-l)*E + l*P, where E = engagement-patch value if the key is in the engagement patch ELSE the BASE config's resolved default, and P = proxy-patch value if the key is in the proxy patch ELSE the BASE default. Thus l=0 reproduces the pure engagement arm and l=1 the pure proxy arm; engagement-only keys relax back to the base default as l->1 and proxy-only keys rise from the base default as l->0. The five fully-resolved patches are tabulated in the NOTES (docs/scenarios/satisfaction_vs_retention-NOTES.md) and regenerated deterministically by scripts/run_phase21_b.sh.
>
> EXPECTED SIGNATURE: retention_7d and sessions_per_user_per_day highest near l=0 and declining toward l=1; welfare.mean_immediate_satisfaction and session_health.mean_session_utility (U_s) lowest near l=0 and rising toward l=1; arch_comfort share shifting across l. A 'knee' where welfare gains flatten marks the efficient trade-off point.
>
> VERDICT CRITERIA: FRONTIER DEMONSTRATED iff engagement/retention and welfare move in OPPOSITE directions across l (negative rank correlation between retention_7d and mean satisfaction over the 5 arms); report the knee. Otherwise report 'no trade-off observed / co-moving'.
>
> ARMS: lambda00, lambda25, lambda50, lambda75, lambda100 -> results/phase21/satisfaction_vs_retention/<arm>.

## Notes

- **No aggregate score** (D22 / V2 TDD §6): every group above (engagement, hidden welfare, session health, recommendation quality, long-term, ecosystem) is reported SEPARATELY; this script never combines them into one number, and neither do plot_results.py's Phase 21 additions.

- **Frozen-schema reads only** (contracts §6): every key this script reads is quoted verbatim from contracts §2 (ecosystem) / §3 (longterm_metrics.csv's trailing column) / §5 (long_term, inherited from Phase 20) / the results_writer.cpp-verified V2 §6 group keys -- no candidate-path guessing. A block genuinely absent for an arm (e.g. a control arm run with a different gate set) renders as a clearly labeled n/a cell, never a crash or a silently-guessed alternate key.

- **Per-day mini-table method**: first/middle/final SIMULATED DAY over the UNION of days present in `ecosystem_metrics.csv` and `longterm_metrics.csv` (the two files are gated independently -- contracts §1 requires the event scheduler for the former, §5's P20 gates for the latter -- so an arm with only one of them still gets a partial row).

- **CSV/MD content split**: `<scenario>-comparison.csv` carries only the flat whole-run arms-x-groups table (one row per arm); the per-day mini-table, the verbatim pre-registration quote, and the verdict are `.md`-only (see this script's module docstring).

- **Concurrency-contention caveat**: if arms were run CONCURRENTLY (contracts §4 caps this at 2 simulate processes per package), wall-clock/timing numbers carry cache and memory-bandwidth contention; every other number in this report is deterministic (rng/clock-free, D8/D9) and unaffected by run mode.
